# FAQ

**Is this a real multisig?**
Yes. A spend needs a threshold of members. The difference from a normal Solana multisig is
that the chain shows none of it: no signer list, no threshold, no balance. On-chain the
vault looks like one ordinary wallet.

**How is it private if it is on a public chain?**
Funds sit in a shielded pool. Each note is a Poseidon commitment, spends are authorized by
a threshold signature verified inside a zero-knowledge proof, and the proof is checked
on-chain. Deposits and withdrawals cannot be linked. It is the Zcash shielded model
brought to a Solana team vault.

**Can one member run off with the funds?**
No. Authorization requires a threshold of key shares, and the shares are never combined.
One member cannot produce a valid signature alone.

**Can the Unbridge team take the funds?**
No. The coordinator holds no key shares and cannot sign. The program has no instruction
that sweeps the pool to an arbitrary address. The upgrade authority cannot touch user
funds.

**What can the relayer see?**
The recipient address of a withdrawal, at the moment it submits it. Not the members, not
the amount's origin, not which note was spent. It cannot alter the withdrawal because the
proof binds the outputs.

**What is actually visible on-chain?**
That a deposit or withdrawal happened, and the standard denomination of a deposit. Not who,
not how much is held, not the link between a deposit and a withdrawal. See
[Security](security.mdx).

**Why fixed deposit amounts?**
Uniform denominations are what make deposits indistinguishable. An arbitrary amount would
be a fingerprint that follows you through the pool.

**Is it audited?**
No. The protocol is unaudited and the trusted setup ceremony is ongoing. Do not deposit
more than you are willing to expose to that risk.

**What is the ceremony?**
The zero-knowledge proving system needs a one-time setup. Its circuit-specific phase was
bootstrapped by the project, so today it rests on operator honesty. Anyone can contribute
fresh entropy at [unbridge.dev/ceremony](https://unbridge.dev/ceremony); one honest
external contributor makes the setup sound. This is disclosed rather than hidden.

**Do I need to trust a server to stay online?**
No for safety, yes for convenience. The coordinator only relays ceremony messages and can
be replaced. Your funds do not depend on it. A vault recovers from on-chain data plus your
wallet.

**Is there a token?**
No. The vault holds SOL.

**Where does the code run?**
In your browser. Proving and signing happen on your device; your key share and note
secrets never leave it.

**How do I verify the program is real?**
```bash
solana program show 6ESjwd4u6qW8SP9PtNwNus1hBJTxKViWra91C36RRALu --url mainnet-beta
```
Full check list, including the instruction surface and the disclosed upgrade
authority, is at [Verify it yourself](verify.mdx).

**Why is the client not published as an npm package?**
The vault client contains the FROST implementation, the Groth16 circuit witness
builder, and the deposit/spend transaction constructors. Publishing them as a
package before the trusted-setup ceremony closes would freeze a snapshot that
still depends on the current ceremony randomness. A pinned client release is
scheduled after the ceremony's next-key rotation. Until then, the browser bundle
at [unbridge.dev](https://unbridge.dev) is the reference client. It runs entirely
in your browser and can be inspected in the network tab.

**Is the program upgradable?**
Yes, for now. The upgrade authority is disclosed as
`YZykTqXgx91g2FSXoTh7q46HJnbwEH17jRhbNzbfppf` and can push a new program version.
It cannot invoke a fund-moving instruction that does not exist in the current
binary (see [Verify](verify.mdx)). Freezing the program (setting the authority to
`None`) is scheduled after the ceremony closes so a bug found during it can still
be patched.

**Where can I read the source?**
The on-chain program's bytecode is on Solana; anyone can fetch and disassemble it
from `6ESjwd4u6qW8SP9PtNwNus1hBJTxKViWra91C36RRALu`. The Rust source will be
published alongside the pinned client after the ceremony closes, together with a
reproducible-build script that maps the source to the on-chain data length. Until
then, the ten instruction names and their auth model are in [Verify](verify.mdx).
