# How it works

The lifecycle of value in an Unbridge vault: create, deposit, authorize, withdraw. Every
step runs on the members' own devices; the chain only ever sees a proof.

## 1. Create the vault

The founders pick `t`-of-`n` (for example 2-of-3). Each member runs distributed key
generation: they exchange verification commitments and share contributions through the
coordinator, verify each other's pieces, and independently derive the same group public
key. No machine ever holds the full signing key. The group public key is the vault's
on-chain identity, and it looks like any other Solana address.

## 2. Deposit

A member deposits SOL into the shielded pool. The deposit builds a **note**: a Poseidon
commitment `C = H(amount, owner, blinding)`. The chain appends `C` as a leaf in the
pool's Merkle tree and stores an encrypted output that only the note's holders can read.

On-chain, an observer sees that some address deposited a standard denomination into the
pool. It cannot see who the note belongs to or tie it to a future withdrawal.

Deposits use fixed denominations. Uniform amounts are what let one deposit blend into the
crowd of others; an arbitrary amount would be a fingerprint.

## 3. Authorize a withdrawal

To move funds out, the members run a FROST signing ceremony:

1. **Round one:** each participating member publishes nonce commitments.
2. **Round two:** once a threshold has committed, each member computes a partial signature over the withdrawal message using their key share and the shared nonces.
3. **Aggregate:** the partials combine into one group signature. The group secret is never assembled.

This is asynchronous. A member can approve, close the tab, and the ceremony still
completes once the threshold is reached. No member ever needs to hand their share to
anyone.

## 4. Prove and settle

The initiating member's device builds a Groth16 zero-knowledge proof that asserts, without
revealing any of it:

- the note being spent exists in the Merkle tree,
- its nullifier has not been used,
- the group threshold signature over the withdrawal is valid,
- inputs and outputs balance.

The proof plus the new output notes and a nullifier go to the relayer, which submits the
transaction and pays the fee. The program verifies the proof, records the nullifier, and
releases the funds to the recipient.

The recipient should be a **fresh address**. Because the relayer pays and the proof hides
the source, a fresh destination has no on-chain path back to the vault or the members.
Sending to an address already tied to a member re-links it, which is a usage mistake, not
a protocol one.

## What each party sees

- **Members:** their own share, the vault's balance (via the view key), the ceremony.
- **The chain:** a deposit of a standard denomination; later, a valid proof and a withdrawal to some address. No connection between the two.
- **A chain analyst:** an anonymity set, not a graph.
