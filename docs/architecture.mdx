# Architecture

Unbridge has four moving parts. Only one of them, the on-chain program, is trusted with
correctness. None of them can move a vault's funds alone.

## On-chain program

A Solana program that owns the shielded pool. It holds:

- A **Merkle tree** of note commitments. Every deposit appends a leaf.
- A set of **spent nullifiers**, so a note cannot be spent twice.
- The **Groth16 verifying key** for the spend and deposit circuits.

Its privileged instructions adjust configuration: per-transaction deposit caps
(`update_deposit_limit`, `update_deposit_limit_for_spl_token`), fee rates and margins
(`update_global_config`), the association-set-provider role (`set_asp_authority`,
`push_association_root`), and per-mint tree init for SPL support
(`initialize_tree_account_for_spl_token`). None of them transfers pool funds to a
caller-chosen address. Value only leaves the pool through `deposit`, `transact`, or
`transact_spl`, each of which verifies a Groth16 proof consuming a note the caller can
prove they own. See [`verify.mdx`](verify.mdx) for the full surface.

## Members and key shares

A vault is a `t`-of-`n` group. Each member generates a share of the group signing key
through a **distributed key generation** (dealerless Feldman VSS): every member verifies
every other member's contribution, and the group public key is derived without the full
private key ever existing on any machine.

To spend, a threshold of members run a **FROST** signing ceremony over Baby Jubjub. Each
contributes a partial signature; the partials aggregate into one group signature. The
group secret is never reconstructed, not even momentarily at signing time.

## Coordinator

Members need a way to find each other during a ceremony (exchange commitments and partial
signatures) without being online at the same instant. The coordinator is a small relay
that passes these messages.

It **holds no key shares**. It cannot sign, cannot spend, and cannot recover the group
key. It is a convenience for liveness, not a trusted party for safety. A different
coordinator, or a direct peer channel, would work equally well.

For team vaults it currently also stores an encrypted **view key** so members can display
shared balances. That key can read amounts; it cannot authorize a spend. Removing it from
the coordinator is on the roadmap.

## Relayer

A withdrawal must be submitted by someone who pays the network fee. If a member submitted
it, their wallet would appear next to the withdrawal and break the unlinkability.

The relayer submits the proof-carrying transaction and pays the fee. It is reimbursed from
a prepaid fee inside the withdrawal itself, so it nets out over time. It learns the
recipient address at submit time, and nothing about the members or which note was spent.
It cannot forge or alter a withdrawal: the proof is bound to the exact outputs.

## Durable nonce for team sends

A threshold ceremony can span minutes or longer as members approve at different times. A
normal Solana blockhash expires in about a minute, which would invalidate a signature
gathered slowly. Team sends therefore spend over a **durable nonce account** whose value
does not expire. The nonce account's rent is paid by the member who initiates the first
send, and its authority is the vault key, so the coordinator does not fund or control it.

## Trust summary

| Party | Can it move funds? | What it learns |
|---|---|---|
| On-chain program | Only by verifying a valid proof | Public tree and nullifiers |
| A single member | No (needs threshold) | Their own share |
| Coordinator | No (no shares) | Ceremony messages, encrypted view key |
| Relayer | No (proof-bound) | Recipient at submit time |
| Admin / upgrade authority | No (no sweep instruction) | Nothing private |
